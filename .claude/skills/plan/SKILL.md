你现在处于「Plan 模式」。

这是一个**多轮协作规划过程**。目标不是写一个摘要文档，而是直接产出一份可审阅、可执行的任务合同：

1. **需求文档**：一级设计源，保留主要技术路径、时序、逻辑分块和模块设计
2. **Plan**：合同正文，包含两个视图
   - `View 1: Review`：给人 review 的说明视图
   - `View 2: Issue Contract`：给执行 AI 的完整 issue 合同视图
3. **Issues CSV**：状态投影，只保存执行状态和最小索引信息，不再表达复杂设计

**这套新流程只有两个 skill：**
- `plan`
- `execute-issues`

不要再拆出任何“Plan → Issues”的中间流程，也不要把 issue 拆分留给另一个 skill 二次理解。

## 一、输入规则

1. `$ARGUMENTS` 可以是：
   - 需求文档路径（优先推荐）
   - 纯文本需求描述
2. 如果输入是需求文档路径：
   - 必须先读取该文档
   - 将其视为**主设计源**
   - Plan 只负责：澄清、覆盖、约束、测试、issue 合同，不要重写需求文档里的主要设计
3. 如果输入是纯文本需求：
   - 正常调研并讨论
   - 最终仍然要生成完整的 Review View + Issue Contract + Issues CSV

## 二、阶段流程

### 阶段 A：调研

收到输入后，先做项目调研，不要急于提方案：

1. 若有需求文档，先读取并提取：
   - 核心技术路径
   - 时序和逻辑分块
   - 模块边界
   - 配置 / 数据结构 / 状态机
   - 需要澄清或覆盖的点
2. 从需求文档或需求描述中提取关键词，用 Grep/Glob 定位相关文件和模块
3. 阅读关键代码，理解现有架构、依赖关系和落地约束
4. 向用户汇报调研结果

**调研汇报格式**：

```text
需求理解：<用自己的话复述需求，确认理解是否正确>

设计来源：
- 主需求文档：<如有，写路径>
- 默认沿用的设计：<需求文档中哪些章节/机制将直接作为实现基础>
- 需要在 Plan 阶段澄清/覆盖的点：<哪些设计还需要讨论>

项目现状：
- <关键发现 1>
- <关键发现 2>
- <关键发现 3>

初步思路：
- 方案 A：<简述> — 优点：... / 代价：...
- 方案 B：<简述> — 优点：... / 代价：...
（如果只有一种合理路径，说明为什么其他路径不可行）

测试思路（初步）：
- <单元/集成/接口测试建议>
- <关键场景>
- 测试文件统一放在 `.dev/test/` 目录下

需要和你确认的问题：
1. <边界疑问>
2. <设计覆盖点>
3. <测试或 issue 粒度确认>
```

### 阶段 B：需求讨论与收敛

在用户回应后，继续多轮讨论，直到方案收敛。

**你应该做的**：
- 明确区分：哪些内容来自需求文档、哪些是 Plan 新增澄清、哪些是 Plan 覆盖决策
- 对可能的技术风险提前预警
- 当讨论中出现新信息，主动回到需求文档和代码中验证
- 逐步收敛 issue 粒度，不要等落盘时临时拆分

**你不应该做的**：
- 不要把需求文档机械复制到 Plan
- 不要在关键点未收敛时急于落盘
- 不要让 issue 拆分依赖落盘后的二次猜测

### 阶段 B2：测试边界讨论（必须执行）

在方案收敛后、落盘前，必须先完成测试边界讨论。

输出格式：

```text
测试点清单（草案）：

| 功能点 | 测试场景 | 场景类型 | 预期结果 |
|--------|---------|---------|---------|
| <功能 1> | <场景描述> | 正常/边界/异常 | <预期> |
| ... | ... | ... | ... |

测试文件位置：`.dev/test/test_<csv-slug>.py`

需要确认：
1. <是否有遗漏的测试场景？>
2. <某个边界条件是否需要覆盖？>
```

关键约束：
- 测试文件固定放在 `.dev/test/`
- 文件名固定为 `test_<csv-slug>.py`

### 阶段 C：落盘（用户明确同意后才执行）

落盘阶段必须一次性完成三件事：

1. 生成 Plan 文件
2. 从 Plan 的 Issue Contract 直接映射生成 Issues CSV
3. 校验 Issues CSV

## 三、Plan 文件规范

### 目录与文件名

- Plan：`.dev/plans/YYYY-MM-DD_HH-mm-ss-<slug>.md`
- CSV：`.dev/issues/YYYY-MM-DD_HH-mm-ss-<slug>.csv`

### Frontmatter 必填字段

- `mode`
- `cwd`
- `task`
- `complexity`
- `created_at`
- `requirements_path`（若有）
- `issues_path`
- `contract_version`

### 正文必须包含两个视图

#### `## View 1: Review`
职责：给人 review，记录为什么这样做。

必须包含：
- 任务概述
- 设计来源
- 需求边界
- 现状分析
- 关键决策
- 否决方案
- 执行约束
- 测试策略
- 风险与注意事项
- 参考

#### `## View 2: Issue Contract`
职责：执行合同正文，给执行 AI 直接使用。

每条 issue 必须写清：
- `id`
- `priority`
- `title`
- `summary`
- `design_refs`
- `code_refs`
- `acceptance_criteria`
- `test_cases`
- `constraints`
- `depends_on`
- `notes`

**重要原则**：
- Issue Contract 不是附录，是执行合同正文
- 执行 AI 必须先读对应 issue 条目，再读 `design_refs` 指向的需求文档设计，再读 `code_refs` 指向的代码，最后才开始写实现
- 如果需求文档已经给出主要设计，Issue Contract 只做执行化映射和约束补充，不重新设计

## 四、Issues CSV 规范

### CSV 角色

CSV 只是状态投影，不再承担复杂设计表达。

### 固定表头

使用以下全新 schema：

```text
id,priority,title,refs,dev_state,test_state,owner,notes
```

字段含义：
- `id`：issue 唯一标识，必须与 Plan 的 Issue Contract 一致
- `priority`：`P0|P1|P2`
- `title`：简短标题
- `refs`：最小索引信息，可由 `design_refs + code_refs` 合并映射
- `dev_state`：`未开始|进行中|已完成`
- `test_state`：`未开始|进行中|已完成|失败`
- `owner`：默认留空
- `notes`：执行备注、阻塞、完成时间等

### CSV 生成与校验

使用 `plan/scripts/` 下的脚本：

- 生成：`plan/scripts/generate-csv.py`
- 校验：`plan/scripts/validate-csv.py`

不要再使用任何旧目录下的脚本或模板。
生成 Issues CSV 后，必须校验通过，否则不能继续执行。
校验通过后需要删除此次任务的中间产物，也就是json文件。

## 五、Issue 粒度原则

每条 issue 应当是一个**可独立开发、可独立测试、可独立验收**的最小交付单元。

优先按以下维度拆分：
1. 按模块 / 文件拆
2. 按新建角色拆（模型 / 服务 / 中间件 / 消费者适配）
3. 按层拆（后端 / 前端 / 基础设施）
4. 按阶段拆（基础能力 → 集成 → 测试补全）
5. 如果需求文档已天然分出逻辑块 / 时序块，应优先映射为 issue
6. 如果一条 issue 同时包含“持久化 / 纯函数算法 / 中间件 hook / 主链路接入 / 测试收口”等多个关注点，必须继续拆分，直到每条 issue 成为一个真正的小闭环

### 默认粒度目标

- `simple`：4-6 条 issue
- `medium`：7-20 条 issue
- `complex`：21-40 条 issue
- `large`：41+ 条 issue，根据复杂度自行拆分，上不封顶

**特别要求**：
- complex和large 任务默认先细拆，再允许合并；不要先粗拆后把细分留给执行 AI 在 issue 内部二次脑补
- 如果需求文档已经提供了较完整的技术路径，issue 拆分应尽量映射需求文档中的状态机节点、时序节点、持久化边界、纯函数边界和主链路接入点
- 默认宁可多 2-3 条清晰的小 issue，也不要少几条但每条内部仍需自行再拆一层

## 六、多次调用规则

- 首次调用：开始新的调研和讨论
- 用户说“基于之前的 Plan 调整”：读取已有 Plan，继续讨论，并**同步更新 Plan + CSV**
- 用户说“修改某条 issue”：必须同时更新 Issue Contract 和 CSV
- 用户说“新的需求”：重新开始一份新 Plan

## 七、原则

- 不要过早收敛
- 需求文档优先，Plan 覆盖显式化
- Plan 是合同正文，CSV 是状态投影
- 任何执行所需的重要设计信息，都必须保留在需求文档或 Plan 中，不能只存在对话里
