# 交互引导与决策话术

协作定稿时按需查阅；与 `SKILL.md` 中「协作输出规范」配合使用。

## 决策对话模板（每轮或每个决策点）

```
【决策点】{一句话：正在决定什么}

| 选项 | 要点 | 与宪法/现有代码 |
|------|------|-----------------|
| A | … | … |
| B | … | … |
| C（可选） | … | … |

【推荐】{默认推荐项}，理由：{1～2 句}
【待你确认】请选择 A/B/C，或补充约束（性能/兼容/上线窗口等）。
```

未获用户确认前，不得将某一选项当作已定事实写入拟落盘的 DES 终稿；若必须先写草案，须单列「待确认假设」表。

---

## 设计目标

```
请描述本次设计的目标：
1. 解决什么问题？
2. 达成什么效果？
3. 有什么设计原则需要遵循？

例如：
- 目标：实现用户认证模块，支持邮箱/手机号登录
- 效果：用户可安全登录，登录响应时间 < 200ms
- 原则：遵循最小权限原则、密码不明文存储
```

## 架构设计

```
请描述系统架构：
1. 系统整体架构（单体/微服务/分层）
2. 核心模块及其职责
3. 模块间的调用关系

建议使用 Mermaid 图表描述。
```

## 接口设计

```
请列出需要新增或修改的接口：

| 接口名称 | HTTP 方法 | 路径 | 说明 |
|----------|-----------|------|------|
| 用户登录 | POST | /api/v1/auth/login | 用户登录认证 |

对于每个接口，请提供：
- 请求参数（含类型、必填性）
- 响应结构（与宪法统一响应格式对齐）
- 错误码（与宪法错误码规范对齐）
```

## 数据库设计

```
请描述数据库变更：

1. 新增/修改的表结构
2. 索引设计
3. 迁移脚本命名

表命名示例（遵循宪法规范）：
- 表名：t_user（若宪法约定 t_ 前缀）
- 字段：user_id, created_at（若宪法约定 snake_case）
```

## 宪法一致性自检

```
请检查设计与项目宪法的一致性：

[ ] 接口路径前缀符合宪法约定
[ ] 响应格式符合宪法统一响应结构
[ ] 错误码在宪法约定范围内
[ ] 表/字段命名符合宪法数据库规范
[ ] 测试策略符合宪法测试规范

如有偏离，请在「风险与注意事项」中说明原因。
```

## 测试点

```
请列出测试点，确保可追溯到 REQ 验收标准：

**单元测试点**：
- [ ] 用户登录成功场景
- [ ] 密码错误场景
- [ ] 账号锁定场景

**集成测试点**：
- [ ] 登录后 Token 有效期验证
- [ ] 登录日志正确记录

**E2E 测试场景**（若适用）：
- [ ] 用户完整登录流程
```
