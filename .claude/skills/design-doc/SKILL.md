# 设计文档撰写（DES）

## 目标与边界

### 主目标

通过 `/design`，将 REQ 与需求要点转化为符合 `references/design-template-standard.md` 的 DES，保存到 `docs/design/`，并与项目宪法对齐，供后续 `/dev-plan` 等流程衔接。

### 非目标（不做评审动作）

- 不组织、不代替**设计评审会**，不以「评审结论 / 通过与否」为主交付物。
- 不以**批注/打分他人既有稿件**为主任务；若用户仅需评审，说明本 skill 默认仍产出 DES 协作流程，或引导其用普通对话完成评审诉求。
- 不输出「评审会议纪要」作为替代 DES 的正式产物。

### 协作模式（默认）

- **默认**：多轮对话，每轮聚焦一块（现状与约束、架构、接口、数据、配置等），**先输出决策点与选项**，用户选择或补充后再写入该块内容。
- **定稿落盘**：仅在满足「可定稿标准」或用户明确要求写文件时，将完整 DES 写入 `docs/design/`。
- **例外**：用户**显式**要求「一次性生成完整 DES 草案 / 跳过讨论」时，可一次输出长文，但文首须附 **「已假设的决策」** 表，未决项标为待确认。

## 协作输出规范（强制）

每个决策点（或每一轮）用结构化形式呈现，避免在散文中藏默认选择：

| 元素 | 要求 |
|------|------|
| 决策点 | 一句话说明在决定什么 |
| 可选方案 | 至少 2 项（可含「维持现状」） |
| 对比 | 各方案优缺点、与宪法及现有代码的符合度 |
| 推荐 | 默认推荐项及理由（可标「待你确认」） |
| 待你确认 | 明确请用户选择或补充约束 |

**禁止**：用户未确认前，用「默认替你选」写入终稿且不标注假设。

**鼓励**：用表格或列表呈现选项；复杂时先给「待决策清单」再展开第一轮。

## 输入要求（强制）

需求来源须显式提供，禁止自动扫描推断「要设计什么需求」：

1. **REQ 文件路径**：`docs/requirement/REQ-{YYYYMMDD}-{序号}-{名称}.md` 或等价约定路径。
2. **需求要点文字**：无法指向文件时，在对话中给出可与 REQ 对齐的要点。

未提供时：提示补充「REQ 路径」或「可结构化设计的文字要点」，不得猜测或扫仓库推断目标需求。

## 项目代码分析（重要）

需求明确后须扫描分析项目代码，使方案与现有系统融合。必查：项目结构、技术栈（`pom.xml`/`go.mod`/`package.json` 等）、配置、分层、代码风格、数据库与迁移、接口形态、公共组件。按需查：相关模块、测试、中间件。

分析结果须体现在 DES 的架构、接口、数据库、配置与风险章节；复杂项目可加强「项目现状分析」小节。

## 文档命名与输出

- 格式：`DES-{YYYYMMDD}-{序号}-{名称}.md`（序号建议与关联 REQ 一致；未指定默认 `001`；名称用安全 slug）。
- 路径：`docs/design/DES-{YYYYMMDD}-{序号}-{名称}.md`。
- 文件名冲突：提示调整序号或名称，禁止静默覆盖。

## 工作流程

### 第一阶段：获取输入

确认 REQ 路径或要点；解析 REQ 元信息；标记空缺项。

### 第二阶段：项目代码分析

扫描结构、技术栈、分层、相关实现、约束与可复用点。空项目可简化，仍须对齐宪法（若存在）。

### 第三阶段：加载宪法与对齐检查

读取 `.ktaicoding/CONSTITUTION.md`；设计不得与生效条文冲突，除非在「风险与注意事项」记录偏离与授权。宪法缺失：建议 `/project-init` 或用户声明临时约定并留痕。REQ 与宪法冲突：列出冲突点与可选方案，待裁决后定稿。

### 第四阶段：协作迭代与决策闭环（常态路径）

对模糊、未闭环或与宪法未对齐的项，**分轮引导**；每轮遵守「协作输出规范」。

**引导顺序**：概述与目标 → 架构与模块 → 接口与数据 → 配置与多服务 → 宪法一致性自检 → 测试点 → 风险。

**触发协作**（任一即进入）：REQ 关键处模糊；接口/数据模型/边界未闭环；缺关键决策；测试点无法覆盖验收标准。

**可定稿标准**（全部满足方可默认完整落盘或标为「已批准」草稿）：

- 关联 REQ 清晰；
- 核心接口与数据变更有实质描述；
- 与宪法无未裁决冲突；
- 测试点可追溯到 REQ 验收项。

未满足：继续协作，或产出带「草案」状态与「待确认假设」表的部分章节。

### 第五阶段：生成文档

按模板生成 Markdown，写入 `docs/design/`。模板优先级：`.aicoding/templates/overrides/design.md` 若存在则优先于内置 `references/design-template-standard.md`。

### 第六阶段：增量更新

用户显式指定目标 DES 路径；保留未变章节；以最新 REQ 与宪法更新；维护「变更记录」。

## 标准章节结构

见 `references/design-template-standard.md`（元信息、概述、现状、架构、接口、库表、时序、配置、多服务、业界参考、测试点、风险、变更记录）。不适用章节保留标题并标 N/A 及原因。

## 交互引导与话术

见 `references/guide-prompts.md`（含决策对话模板与各主题示例话术）。

## 错误处理

| 场景 | 处理方式 |
|------|----------|
| 未提供输入 | 提示：请提供 REQ 路径或需求要点描述 |
| REQ 不存在 | 提示路径无效，请确认 |
| 宪法不存在 | 建议 `/project-init` 或声明临时约定 |
| 文件名冲突 | 提示调整序号或名称，禁止覆盖 |
| 输出目录不存在 | 创建 `docs/design/` |
| REQ 与宪法冲突 | 列出冲突点，请用户裁决 |

## 约束

- 禁止自动推断需求；禁止为分配序号而扫描 `docs/design/`。
- 需求明确后必须分析项目代码。
- 不实现 `/req`、`/dev`、`/cr` 等其他技能；不导出非 Markdown。
- 宪法为硬约束，不得静默忽略。
- 非「评审动作」；主产出为协作迭代后的 DES 落盘。

## 与上游衔接

- 输入：`docs/requirement/REQ-*.md` 或需求要点。
- 宪法：`.ktaicoding/CONSTITUTION.md`（若存在）。

## 与下游衔接

DES 路径 `docs/design/DES-*.md` 元信息正确关联 REQ，便于 `/dev-plan` 与追溯。
