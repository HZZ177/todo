# Project Init（项目初始化）

## 目标

通过单一入口命令 `/project-init`，自动识别场景并协助用户建立、补全或修订后端项目宪法文档（CONSTITUTION.md）。宪法文档是后续需求、设计、开发、审查、发布等流程的规范依据。

## 触发方式

- **唯一入口**: `/project-init`

## 场景自动识别

启动后根据仓库状态自动进入下列之一：

### A. 新项目初始化

**识别条件**: `.ktaicoding/CONSTITUTION.md` 不存在，且代码体量极少或为空（空仓库/仅脚手架）。

**执行流程**:
1. 欢迎用户并说明将进行新项目初始化
2. 通过交互问答收集项目信息：
   - 运行时与语言版本、包管理/构建工具（Maven、Gradle、pip、Poetry、go.mod 等）
   - 服务形态（单体后端、多模块、微服务边界）
   - 对外 API 形态（REST 等）、错误与分页等约定倾向
   - 数据层（关系库/NoSQL 等）与迁移策略
   - 安全与合规（认证授权、敏感数据、日志与密钥）
   - 测试策略（单测、集成测试、契约/接口测试）
3. 基于收集的信息，生成 `.ktaicoding/CONSTITUTION.md`
4. 展示生成的宪法文档，请用户确认或修改

### B. 旧项目初始化

**识别条件**: 无 CONSTITUTION.md 或文件明显不完整，但仓库已有可分析的后端代码与工程文件。

**执行流程**:
1. 欢迎用户并说明将分析现有代码生成宪法
2. 扫描代码库，提取后端约定：
   - 目录与模块/分层结构
   - 构建与依赖声明（pom.xml、build.gradle、requirements.txt、pyproject.toml、go.mod 等）
   - 命名、注释、包结构与分层习惯
   - 数据库相关（迁移脚本、ORM 使用方式）
   - HTTP API 风格（路由前缀、响应包装、错误码）
   - 后端测试布局与覆盖情况
3. 形成宪法草案
4. 展示草案，请用户确认或修改后落盘为 `.ktaicoding/CONSTITUTION.md`

### C. 补充完善

**识别条件**: 已存在 `.ktaicoding/CONSTITUTION.md`，需增补章节、修订条文或根据代码/工程变更同步更新。

**执行流程**:
1. 读取现有宪法文档
2. 对照用户输入与当前仓库状态，识别：
   - 缺失章节
   - 模糊条文
   - 与代码不一致处
3. 列出拟修订条目
4. 用户确认后更新正文，记录版本变更

## 宪法文档结构

CONSTITUTION.md 应覆盖以下11个章节（后端视角）：

1. **元信息**: 项目名称、类型、后端技术栈、服务架构概述
2. **技术约束**: 运行环境、构建与打包、安全、性能（后端）
3. **代码规范**: 目录与模块、命名、注释、风格、分层约定
4. **数据库规范**: 表/字段命名、索引、迁移约定（若适用）
5. **接口规范**: 对外 API（如 REST）、统一响应与错误码、分页等（若适用）
6. **测试规范**: 测试优先、单元/集成/接口或契约测试等
7. **内置约定**: 公共模块、二方库、通用工具与中间件使用原则
8. **文档规范**: README、接口说明、变更记录
9. **Git 规范**: 分支与提交信息
10. **发布规范**: 版本号与发布前检查
11. **历史变更**: 宪法版本记录

## 版本治理规则

### 版本号规则（语义化）

- **MAJOR**: 破坏性变更（删除规范、重大调整）
- **MINOR**: 新增规范或约束
- **PATCH**: 澄清、纠错、说明补充

### 影响分析

发布新版宪法前，提供「影响分析」语义：对照新条文，提示可能不合规的后端代码区域或待调整清单。

## 产物与存放位置

- **目录**: `.ktaicoding/`（项目根目录下）
- **主产物**: `.ktaicoding/CONSTITUTION.md`（项目宪法/后端规范总纲）
- 若目录不存在，由 Skill 创建

## 约束

- 本版本仅面向后端项目（服务、API、数据层等），不包含前端工程与前端规范
- 不实现需求/设计/发布等附录文档模板库
- 不负责实现 /req、/design、/dev 等命令
- 宪法文档是规范依据，修订时需用户确认
