# Requirement Clarify（需求澄清）

## 目标

通过 `/req` 或 `/req --simple` 命令，将用户提供的草稿目录或文字描述整理成规范化的需求文档（REQ），保存到 `docs/requirement/` 目录。文档可作为后续 `/design` 流程的输入。

## 触发方式

- **标准版**: `/req` — 生成完整的需求文档
- **简化版**: `/req --simple` — 生成简化版需求文档，适用于小需求

## 输入要求（强制）

**禁止自动扫描仓库**。用户必须显式提供以下之一：

1. **需求草稿目录**：用户指定的目录路径，技能仅读取该目录下用户指明的文件
2. **需求文字描述**：直接在对话中给出的 PRD/要点/口述内容

**未提供输入时**：提示用户补充「草稿目录」或「文字描述」，不得猜测或扫描仓库。

## 文档命名规则

### 格式
```
REQ-{YYYYMMDD}-{序号}-{名称}.md
```

- `{YYYYMMDD}`：8 位日期（公历）
- `{序号}`：当日流水序号，用户显式指定；未指定时默认 `001`，若已存在则提示用户指定新序号
- `{名称}`：需求简称，做安全文件名处理（小写/连字符 slug、去非法字符）

### 输出文件
- **标准版**: `docs/requirement/REQ-{YYYYMMDD}-{序号}-{名称}.md`
- **简化版**: `docs/requirement/REQ-{YYYYMMDD}-{序号}-{名称}-简化版.md`

### 文件名冲突
若已存在相同文件名，提示用户调整序号或名称，禁止静默覆盖。

## 工作流程

### 第一阶段：获取输入

1. 确认用户是否提供了草稿目录或文字描述
2. 若未提供，提示用户补充
3. 解析已有内容，映射到模板章节
4. 标记空缺项

### 第二阶段：引导完善（强制）

对空缺、模糊、占位内容进行分轮交互引导。

**引导顺序**：
1. 用户故事（强引导）
2. 目标用户与背景（强引导）
3. 功能与流程（强引导）
4. **本期边界：本期范围 / 本期不做**（强引导，重点覆盖）
5. 历史关联需求（可选）
6. 行业与竞品（可后置，可选）
7. 验收与 DoD（强引导）

**引导方式**：
- 每块给出简短说明 + 具体问题示例
- 对业务流程追问主路径与异常分支
- 对「本期不做」要求列出至少 N 条不做的场景或功能

**最低可评审标准**：
- 清晰的用户故事
- 可列出的功能点与优先级
- 明确的本期范围与本期不做
- 可执行的验收标准

### 第三阶段：生成文档

按模板生成 Markdown 文件，保存到 `docs/requirement/`。

### 第四阶段：增量更新

再次执行时：
1. 用户须显式指定待更新的目标文件路径
2. 保留未变更的章节内容
3. 仅更新用户修正的部分
4. 更新文末「变更记录」

## 文档结构

### 标准版章节（/req）

详见 `references/requirement-template-standard.md`。

必须包含：
1. **元信息**：文档编号、创建日期、负责人、状态
2. **一、用户故事**：角色/功能/目的
3. **二、目标用户**：用户类型、描述、核心诉求
4. **三、需求背景**：业务背景、要解决的问题、预期收益
5. **四、需求详细说明**：功能列表（序号、功能点、优先级、说明）、业务流程
6. **五、本期边界**：本期范围、本期不做（重点！）
7. **六、历史相关需求**：关联 REQ 编号、关系、说明
8. **七、行业情况**：业界参考、竞品分析
9. **八、验收要求**：验收标准、完成定义 DoD
10. **九、附录**：变更记录表

### 简化版章节（/req --simple）

详见 `references/requirement-template-simple.md`。

必须包含：
1. **元信息**：文档编号、创建日期、负责人、状态
2. **一、用户故事**：角色/功能/目的
3. **二、功能说明**：功能描述、功能列表（含本期边界）
4. **三、验收标准**：可勾选列表或条目
5. **变更记录**

## 模板覆盖

若项目存在 `.aicoding/templates/overrides/requirement.md`，优先使用该模板。

## 引导话术示例

### 用户故事引导

```
请描述用户故事，格式为：
"作为一名 [角色]，我希望 [功能]，以便 [目的]"

例如：
"作为一名电商用户，我希望能够查看订单历史，以便追踪我的购买记录"
```

### 本期边界引导

```
本期边界是防止范围蔓延的关键。请明确：

**本期范围**：本期要交付的功能/场景
- 例如：用户登录、商品浏览、下单支付

**本期不做**：明确排除的功能/场景
- 例如：会员积分、优惠券、多语言支持

请列出至少 3 条「本期不做」的内容。
```

### 验收标准引导

```
验收标准应可量化、可测试。请按以下格式：

- [ ] 功能点 A：预期行为描述
- [ ] 功能点 B：预期行为描述

例如：
- [ ] 用户可使用邮箱注册账号
- [ ] 注册后收到确认邮件
```

## 错误处理

| 场景 | 处理方式 |
|------|----------|
| 未提供输入 | 提示：「请提供草稿目录路径或需求文字描述」 |
| 目录不存在 | 提示：「目录 {path} 不存在，请确认路径」 |
| 文件名冲突 | 提示：「文件已存在，请调整序号或名称」 |
| 输出目录不存在 | 创建 `docs/requirement/` 目录 |

## 约束

- 不自动扫描仓库或 `docs/requirement/` 目录
- 不为自动分配序号而扫描目录
- 不实现 `/design`、`/dev` 等其他技能
- 不导出非 Markdown 格式

## 与下游衔接

生成的 REQ 文档路径约定为 `docs/requirement/REQ-{YYYYMMDD}-{序号}-{名称}.md`，元信息中的文档编号与文件名一致，便于 `/design` 流程引用。
