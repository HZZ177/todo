# 代码合规审查

## 目标

以项目宪法（`.ktaicoding/CONSTITUTION.md`）为唯一审查基准，**动态**解析宪法结构，对代码进行多维度并行合规审查，产出结构化审查报告，帮助团队在合并前发现并修正不合规代码。

**宪法外禁报（强制）**：报告中的每一条 finding 都必须能在当次审查所使用的宪法原文（含对应章节/子节）中找到**直接依据**。不得写入下列内容，即使它们在其他语境下是「好建议」：

- PEP 8、语言惯用法、通用整洁代码、DRY、圈复杂度、性能微优化等**未写入宪法**的规范
- 行业惯例、个人偏好、团队口头约定（未落进宪法文件的）
- 「更优雅」「更可读」「建议重构」等**宪法未要求**的改进意见
- 宪法未规定的注释密度、文档详略、测试覆盖率具体阈值

若某现象在宪法中**完全没有对应条文**，则**不列为问题**；可在审查小结中一句话说明「以下方面宪法未规定，本审查未覆盖」即可，**不得**虚构 severity 或凑数条目。主会话聚合各子任务结果时，若某条缺少可核验的宪法原文依据，应丢弃该条，不写入落盘报告。

## 触发方式

- **唯一入口**: `/code-compliance-review`

## 前置条件（强制）

每次审查启动时，**必须**重新读取 `.ktaicoding/CONSTITUTION.md`（使用 Read 工具或等价方式，**禁止**依赖会话记忆、摘要或旧版缓存）。

- **宪法存在** → 以**当前文件全文**为唯一审查基准；从文档**元信息（版本/日期）**或历史变更表格首行提取 `constitution_version` 写入报告
- **宪法不存在** → **立即终止**，输出以下信息并结束：

```text
[code-compliance-review] 审查终止

未找到项目宪法文件 .ktaicoding/CONSTITUTION.md。
代码合规审查必须以项目宪法为基准，无宪法则无审查依据。

请先运行 /project-init 生成项目宪法，再发起代码审查。
```

## 三种审查模式

### 模式 A：Issues 关联审查（推荐）

用户提供 issues CSV 路径，审查范围 = CSV 中 `refs` 字段涉及的文件 + `git diff` 变更文件。

**触发**: `/code-compliance-review <issues CSV 路径>`

**范围确定流程**：
1. 读取 issues CSV（默认路径 `.dev/issues/`），提取所有行的 `refs` 字段，解析出涉及的文件路径
2. 定位与该 CSV 同 basename 的 Plan 文件（`.dev/plans/<same-basename>.md`），从 Issue Contract 中提取 `code_refs`
3. 运行 `git diff --name-only` 获取工作区变更文件列表
4. 合并上述文件集合，去重后作为审查范围
5. 若合并后文件列表为空，提示用户确认

### 模式 B：全量项目审查

对整个项目代码做宪法合规全扫描。适用于项目初期建立基线、或重大规范变更后的全面排查。

**触发**: `/code-compliance-review --full`

**范围确定流程**：
1. 扫描项目代码目录（排除 `.git/`、`node_modules/`、`.venv/`、`__pycache__/`、`.dev/` 等非业务目录）
2. 按宪法中「目录结构」相关章节识别业务代码范围
3. 文件列表即为审查范围

### 模式 C：二次审查（基于上次报告）

用户修复了上次审查报告中的问题后，发起复查。以上次报告为基线，聚焦验证修复情况并检查新引入的问题。

**触发**: `/code-compliance-review --recheck <上次报告路径>`

**范围确定流程**：
1. 读取指定的上次审查报告
2. 提取报告中所有「严重」和「重要」级别的未修复问题及其涉及的文件
3. 运行 `git diff --name-only` 获取自上次审查以来的变更文件
4. 合并上述文件集合，作为本次审查范围
5. 子任务审查时额外携带上次报告中的问题清单，要求逐条验证修复状态

## 审查流程（所有模式通用）

### 第一步：加载宪法并动态解析审查维度

读取 `.ktaicoding/CONSTITUTION.md` 全文后，**动态**解析其章节结构，按以下规则将章节分组为审查维度。

#### 1.1 解析宪法章节

扫描宪法中所有顶级章节（`## N. 标题` 格式）及其子章节，构建完整的章节清单。示例输出：

```text
已识别宪法章节：
  §1 元信息
  §2 技术约束（含 §2.1–§2.4）
  §3 代码规范（含 §3.1–§3.12）
  §4 数据库规范（含 §4.1–§4.5）
  §5 接口规范（含 §5.1–§5.6）
  §6 测试规范（含 §6.1–§6.4）
  §7 内置约定（含 §7.1–§7.2）
  §8 文档规范
  §9 Git 规范
  §10 发布规范
  §11 历史变更
```

#### 1.2 章节分组为审查维度

使用以下**分组策略**将章节映射到审查维度。分组策略关注章节的**语义内容**而非固定编号——宪法章节编号可能随版本变化，但语义类别相对稳定：

| 维度 ID | 维度名称 | 分组规则（按语义匹配，不锁定章节号） |
|---------|---------|--------------------------------------|
| DIM-CODE | 代码规范 | 涵盖「代码规范」「代码风格」「命名规范」「分层约定」「前端规范」等相关章节 |
| DIM-DB | 数据库规范 | 涵盖「数据库」「表设计」「索引」「迁移」等相关章节 |
| DIM-API | 接口规范 | 涵盖「接口」「API」「请求」「响应」「分页」「前端响应约定」等相关章节 |
| DIM-TEST | 测试规范 | 涵盖「测试」「单元测试」「集成测试」相关章节，以及代码规范中与测试策略相关的子章节 |
| DIM-CONV | 内置约定与流程 | 涵盖「内置约定」「公共模块」「中间件」「文档规范」「Git 规范」「发布规范」等约定性章节 |
| DIM-SEC | 安全与技术约束 | 涵盖「技术约束」「安全」「部署」「运行环境」等相关章节 |

**分组执行规则**：

1. **读取宪法后现场分组**：根据每个章节的实际标题和内容，判断其归属的维度。同一章节只归入一个维度（例外：某些子章节可能跨维度关联，如「前端测试策略」同时与 DIM-CODE 和 DIM-TEST 相关——此时在 DIM-TEST 中作为主要审查点，DIM-CODE 可标注参考）
2. **新增章节自动归入**：如果宪法新增了上表未直接提及的章节，根据语义归入最匹配的维度。若完全无法归入现有维度，创建新的 `DIM-EXTRA` 维度
3. **元信息和历史变更不审查**：「元信息」「历史变更」类章节不产生代码审查维度，仅用于提取版本号
4. **每个维度携带的 `{CONSTITUTION_SECTION}` 必须是该维度对应章节的完整原文**（从宪法中切出，不手写删减、不做摘要）

#### 1.3 输出维度分组结果

分组完成后，向用户输出维度映射摘要供确认：

```text
[code-compliance-review] 维度映射（基于宪法 vX.X.X）

  DIM-CODE  ← §3 代码规范（含 §3.1–§3.12）
  DIM-DB    ← §4 数据库规范（含 §4.1–§4.5）
  DIM-API   ← §5 接口规范（含 §5.1–§5.6）
  DIM-TEST  ← §6 测试规范 + §3.11 前端测试策略
  DIM-CONV  ← §7 内置约定 + §8 文档规范 + §9 Git 规范 + §10 发布规范
  DIM-SEC   ← §2 技术约束
  [跳过]    ← §1 元信息、§11 历史变更（非代码审查维度）
```

（上述为示例，实际输出根据当次解析结果动态生成）

### 第二步：智能裁剪维度

并非所有维度都适用于每次审查。根据审查范围内的文件类型智能裁剪：

- 审查范围内无数据库模型/迁移文件 → 跳过 DIM-DB
- 审查范围内无后端路由/Schema 实现，且无前端 API 封装 → 跳过 DIM-API
- 审查范围内无测试文件 → DIM-TEST 审查「是否应按宪法测试规范补测试或抽离可测逻辑」
- 审查范围仅前端 → DIM-SEC 侧重与前端相关的约束部分
- 审查范围无文档/配置文件 → DIM-CONV 中的文档与发布规范可标记「范围外未审」
- 全量审查模式下（`--full`）→ 所有维度均执行

裁剪完成后，向用户输出审查计划摘要：

```text
[code-compliance-review] 审查计划

宪法版本: <从宪法中提取>
审查模式: <A/B/C>
审查范围: <文件数量> 个文件
<若模式 C> 基线报告: <上次报告路径>

启用维度:
  [启用] DIM-CODE -- 代码规范（XX 个文件）
  [启用] DIM-API  -- 接口规范（XX 个文件）
  [启用] DIM-TEST -- 测试规范（XX 个文件）
  [跳过] DIM-DB   -- 跳过（范围内无数据库相关文件）
  ...

即将派发 N 个审查子任务并行执行。
```

### 第三步：派发子任务并行审查（强制）

**每个启用的维度必须派发独立的子任务执行审查**，不允许在主会话中直接审查代码。

对每个启用的维度，使用 Task 工具派发子任务，按 `agents/compliance-reviewer.md` 模板填充以下上下文：

- `{DIMENSION_ID}` — 维度 ID（如 DIM-CODE）
- `{DIMENSION_NAME}` — 维度名称
- `{CONSTITUTION_SECTION}` — 该维度对应的宪法章节**完整原文**（不是摘要，不是检查清单——是从宪法切出的一整段或多段原文。若维度横跨多个章节，拼接所有相关章节并保留章节标题）
- `{FILE_LIST}` — 本维度需审查的文件路径列表
- `{REVIEW_MODE}` — 审查模式（A/B/C）
- `{PREVIOUS_ISSUES}` — 仅模式 C：上次报告中本维度的问题清单；其他模式为空

**并行策略**：所有子任务在同一轮工具调用中派发，最大化并行度。

**子任务配置**：
- `subagent_type`: `generalPurpose`
- `model`: 默认使用 `fast` 以控制成本；若用户要求深度审查可切换更强模型
- `readonly`: `true`（审查不修改代码）

### 第四步：聚合审查结果

所有子任务返回后，汇总各维度审查结果：

1. **过滤非宪法依据**：丢弃任何无法在当次已加载的 `.ktaicoding/CONSTITUTION.md` 原文中定位到对应条文的 finding（含 `constitution_text` 与宪法实际内容不符、或引用章节在宪法中不存在的条目）
2. 按严重程度分级聚合：严重 → 重要 → 轻微
3. 同一文件的多维度问题就近合并展示
4. 统计各级别问题数量
5. 若为模式 C（二次审查），额外生成修复状态对比：
   - [已修复]
   - [未修复]
   - [新引入]

### 第五步：生成审查报告并落盘

审查报告**必须**写入文件，路径：`.dev/reviews/REVIEW-<YYYY-MM-DD>-<seq>-<slug>.md`

- `<seq>`：当日序号，从 001 开始
- `<slug>`：简短描述（如 `memory-queue`、`full-scan`、`recheck-001`）

报告结构见下方「审查报告格式」。

落盘后向用户输出摘要：

```text
[code-compliance-review] 审查完成

报告已生成: .dev/reviews/REVIEW-<YYYY-MM-DD>-<seq>-<slug>.md

审查统计:
  严重: X 项
  重要: X 项
  轻微: X 项
  总计: X 项

<若模式 C>
修复状态:
  已修复: X 项
  未修复: X 项
  新引入: X 项

请查阅报告了解详情。修复后可运行:
/code-compliance-review --recheck .dev/reviews/REVIEW-<YYYY-MM-DD>-<seq>-<slug>.md
```

### 报告详细度要求（强制）

审查报告中每一条问题的记录**必须足够详细**，让未参与审查的开发者仅凭报告就能准确理解问题所在并完成修复。具体要求：

1. **现状描述必须具体**：不能只写"未使用统一 HTTP 客户端"，必须写明当前代码用了什么（如"直接 `import httpx` 并调用 `httpx.AsyncClient()`"），在哪个函数/方法中（如"在 `run_native_memory_extraction()` 方法的第 417-465 行"），相关的代码片段是什么样的。
2. **宪法依据必须引用原文**：不能只写"§3.5"，必须写明条文内容（如"§3.5 规定：统一使用 `common.core.http_client` 作为异步 HTTP 客户端，禁止直接使用 httpx/aiohttp"）。
3. **[严重] 级别问题必须包含影响范围**：说明该问题可能导致的具体后果（如安全风险、数据不一致、运行时错误等）。
4. **修复建议必须可操作**：不能只写"改为使用统一封装"，应给出具体修改方案（如"将 `async with httpx.AsyncClient() as client:` 替换为 `from common.core.http_client import async_client`，并使用 `async_client.post(...)` 调用"）。尽量给出修复前后的代码对比示例。
5. **[轻微] 级别可以适当简略**，但仍须包含文件、行号、**与宪法原文一致的**依据和具体描述，不能只用一句话概括；且该条必须属于宪法已规定的事项。
6. **禁止将多个不同问题合并为一条记录**。每个问题独立成条，即使它们在同一个文件中。

## 子任务审查模板

每个子任务的详细指令见 `agents/compliance-reviewer.md`。模板中定义了：

- 审查者的角色和职责
- 基于宪法原文自行推导审查检查清单的方法论
- 问题发现的输出格式（要求详细记录）
- 严重程度的判定标准

## 与其他 Skill 的协作关系

```text
/project-init              → 生成 .ktaicoding/CONSTITUTION.md（审查基准）
          ↓
/dev-plan                  → 生成 Plan + Issues CSV（任务合同）
          ↓
/dev-plan-execute          → 执行 Issue，编写代码
          ↓
/code-compliance-review    → 基于宪法审查代码合规性 ← 你在这里
          ↓
修复 → /code-compliance-review --recheck → 验证修复 → 合并
```

**典型工作流**：

1. 通过 `/project-init` 建立项目宪法
2. 通过 `/dev-plan` 生成开发计划与 Issues CSV
3. 通过 `/dev-plan-execute` 执行开发任务
4. 开发完成后，运行 `/code-compliance-review <issues CSV 路径>` 进行合规审查
5. 根据审查报告修复问题
6. 运行 `/code-compliance-review --recheck <报告路径>` 验证修复
7. 全部通过后合并分支

## 约束

- 本技能是只读审查工具，**不修改任何业务代码**
- 审查基准**仅限**项目宪法**条文本身**，不基于主观偏好、外部规范或未写入宪法的「最佳实践」；报告**不得**包含宪法未规定的代码问题
- 子任务审查是强制的，主会话禁止直接审查代码内容
- 审查报告必须落盘，不能仅输出到对话中
- 二次审查必须引用上次报告，不能凭空声称问题已修复
- 报告中**禁止使用任何 emoji 符号**（如表情、图标等），所有标记使用中文方括号形式，如 `[严重]`、`[已修复]`
- 报告中**尽量使用中文**，避免不必要的英文术语
